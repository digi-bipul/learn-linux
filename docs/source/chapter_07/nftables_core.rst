=======================================
Core Firewalls with nftables
=======================================

Every packet that enters or leaves a Linux system passes through the kernel's
**netfilter** framework. Netfilter is the engine; **nftables** is the
userspace configuration tool that programs that engine. Together, they
constitute the Linux packet filtering and mangling subsystem.

This section covers nftables from the ground up: the architecture (tables,
chains, rules), the verdict system (accept, drop, reject), connection tracking
for stateful firewalling, Network Address Translation (NAT), and the important
migration path from the legacy ``iptables`` framework.

The Big Picture: netfilter Hooks
=======================================

The Linux kernel's network stack has five **netfilter hooks** — points in the
packet traversal path where firewall rules can intercept traffic:

::

    Incoming Packet
          |
          v
    [PREROUTING]  ──→  Routing Decision  ──→  [FORWARD]
          |                       |
          v                       v
    [INPUT]             [OUTPUT]                [POSTROUTING]
          |                       |
          v                       v
     Local Process       Local Process            Outgoing Packet

* **NF_INET_PRE_ROUTING** — All incoming packets hit this hook before the
  routing decision. Used for DNAT (destination NAT).
* **NF_INET_LOCAL_IN** — Packets destined for the local host (after routing).
* **NF_INET_FORWARD** — Packets routed through the host (not destined for the
  host itself).
* **NF_INET_LOCAL_OUT** — Packets generated by the local host.
* **NF_INET_POST_ROUTING** — All outgoing packets (after routing). Used for
  SNAT / masquerading.

nftables rules are attached to these hooks via **chains**.

Tables, Chains, and Rules
================================

nftables organises configuration into a three-level hierarchy:

+----------------+----------------------------------------------------------+
| Level          | Description                                              |
+================+==========================================================+
| **Table** | A namespace containing chains. A table is associated     |
|                | with an **address family** (ip, ip6, inet, arp, bridge). |
|                | Use ``inet`` for dual-stack (IPv4 + IPv6) rules.         |
+----------------+----------------------------------------------------------+
| **Chain** | An ordered list of rules. Chains can be **base chains** |
|                | (attached to a netfilter hook) or **regular chains** |
|                | (jump targets for organisation).                         |
+----------------+----------------------------------------------------------+
| **Rule** | A single filtering instruction: match criteria + verdict.|
+----------------+----------------------------------------------------------+

**Example — a minimal nftables ruleset:**

.. code-block:: bash

    #!/usr/sbin/nft -f

    # Define a table for the inet (IPv4+IPv6) family
    table inet filter {

        # A base chain attached to the INPUT hook
        chain input {
            type filter hook input priority filter; policy drop;

            # Allow loopback traffic
            iif "lo" accept

            # Allow established and related connections
            ct state established,related accept

            # Allow SSH on port 22
            tcp dport 22 accept

            # Log and drop everything else
            log prefix "nftables-drop: " drop
        }

        # A base chain for forwarding
        chain forward {
            type filter hook forward priority filter; policy drop;
        }

        # A base chain for output
        chain output {
            type filter hook output priority filter; policy accept;
        }
    }

**Key syntax elements explained:**

* ``table inet filter`` — Creates a table named ``filter`` for the ``inet``
  (both IPv4 and IPv6) family.
* ``chain input { type filter hook input priority filter; policy drop; }`` —
  A base chain named ``input``, attached to the ``input`` hook, with priority
  ``filter`` (standard value 0). The default **policy** is ``drop`` —
  any packet that falls through all rules is dropped.
* ``iif "lo" accept`` — If the input interface is ``lo`` (loopback),
  immediately accept.
* ``ct state established,related accept`` — Stateful rule: accept packets
  that belong to an established connection or are related to one.
* ``tcp dport 22 accept`` — Accept TCP packets destined for port 22.
* ``log prefix "nftables-drop: " drop`` — Log the packet and then drop it.

The Verdicts
===================

Every nftables rule ends with a **verdict** — the action to take:

+----------------+----------------------------------------------------------+
| Verdict        | Effect                                                   |
+================+==========================================================+
| ``accept``     | Allow the packet through this chain immediately.         |
+----------------+----------------------------------------------------------+
| ``drop``       | Silently discard the packet. No response is sent to the  |
|                | sender.                                                  |
+----------------+----------------------------------------------------------+
| ``reject``     | Discard the packet and send an ICMP error back to the    |
|                | sender (``reject with icmp-port-unreachable``, etc.).    |
+----------------+----------------------------------------------------------+
| ``queue``      | Pass the packet to a userspace application via           |
|                | ``nfqueue`` (e.g., for deep packet inspection).          |
+----------------+----------------------------------------------------------+
| ``jump``       | Continue processing in another (regular) chain.          |
+----------------+----------------------------------------------------------+
| ``goto``       | Like ``jump`` but does not return — processing continues |
|                | in the target chain.                                     |
+----------------+----------------------------------------------------------+
| ``return``     | Return from a ``jump``-ed chain to the calling chain.    |
+----------------+----------------------------------------------------------+

**Drop vs. Reject:** ``drop`` gives no feedback to the sender, which can cause
timeouts. ``reject`` sends back an ICMP "port unreachable" or "host
unreachable" message, allowing the sender to fail fast. In security contexts,
``drop`` is often preferred to avoid revealing that a host exists.

Stateful Filtering with Connection Tracking
==================================================

Connection tracking (``conntrack``) is the kernel's ability to remember the
state of each network connection. nftables uses ``ct state`` to match on
connection states:

+----------------+----------------------------------------------------------+
| State          | Meaning                                                  |
+================+==========================================================+
| ``new``        | The packet is starting a new connection (e.g., SYN in    |
|                | TCP).                                                    |
+----------------+----------------------------------------------------------+
| ``established``| The packet belongs to an existing connection (e.g.,      |
|                | subsequent packets in a TCP stream, or return traffic).  |
+----------------+----------------------------------------------------------+
| ``related``    | The packet is associated with an existing connection but |
|                | belongs to a different stream (e.g., FTP data channel,   |
|                | ICMP errors).                                            |
+----------------+----------------------------------------------------------+
| ``invalid``    | The packet could not be identified — should be dropped.  |
+----------------+----------------------------------------------------------+

The classic stateful firewall pattern:

.. code-block:: nft

    chain input {
        type filter hook input priority filter; policy drop;

        # Accept all loopback
        iif "lo" accept

        # Let established/related through
        ct state established,related accept

        # Drop invalid packets
        ct state invalid drop

        # For new connections, check specific services
        ct state new tcp dport 22 accept
        ct state new tcp dport 80 accept
        ct state new tcp dport 443 accept

        # Everything else is dropped by policy
    }

This pattern is **stateful** because once a connection is established, return
traffic is automatically allowed — you do not have to write rules for ephemeral
ports or response packets.

NAT with nftables
========================

Network Address Translation rewrites the source or destination address of
packets as they cross a routing boundary. nftables handles NAT through
dedicated chain types.

**Source NAT (SNAT) — Masquerading:**

Used on a router/gateway to translate private (RFC 1918) source addresses to
the router's public IP.

.. code-block:: nft

    table inet nat {
        chain postrouting {
            type nat hook postrouting priority srcnat; policy accept;

            # Masquerade traffic from the internal network out through eth0
            ip saddr 10.0.0.0/24 oif "eth0" masquerade
        }
    }

``masquerade`` is a special form of SNAT that automatically uses the IP
address of the outgoing interface. Use ``masquerade`` when the public IP may
change (e.g., a DHCP-assigned address on a home router). For static public
IPs, you can use explicit SNAT:

.. code-block:: nft

    ip saddr 10.0.0.0/24 oif "eth0" snat to 203.0.113.10

**Destination NAT (DNAT) — Port Forwarding:**

Used to redirect incoming traffic from a public IP to a specific internal host.

.. code-block:: nft

    table inet nat {
        chain prerouting {
            type nat hook prerouting priority dstnat; policy accept;

            # Forward HTTP traffic to internal web server
            iif "eth0" tcp dport 80 dnat to 10.0.0.10:80
        }
    }

**Key distinction:** SNAT rewrites the source IP; DNAT rewrites the destination
IP. SNAT happens in ``postrouting``; DNAT happens in ``prerouting``.

Managing nftables at Runtime
====================================

.. code-block:: bash

    # List all rulesets (human-readable)
    sudo nft list ruleset

    # List a specific table
    sudo nft list table inet filter

    # Flush (delete) all rules in a table
    sudo nft flush table inet filter

    # Delete a table entirely
    sudo nft delete table inet filter

    # Add a rule (to a specific chain, at a specific position)
    sudo nft add rule inet filter input tcp dport 443 accept

    # Insert a rule at the beginning
    sudo nft insert rule inet filter input tcp dport 8080 accept

    # Delete a rule by handle (get handles with -a)
    sudo nft -a list ruleset
    sudo nft delete rule inet filter input handle 42

**Saving and restoring:**

nftables rules are **not automatically persistent**. You must save them and
load them at boot:

.. code-block:: bash

    # Save the current ruleset to a file
    sudo nft list ruleset > /etc/nftables.conf

    # Restore from file
    sudo nft -f /etc/nftables.conf

    # On most distributions, enable the nftables service for auto-load
    sudo systemctl enable --now nftables

    # The service reads /etc/nftables.conf by default

The Migration from iptables to nftables
===============================================

**Historical context:** For nearly two decades, Linux packet filtering was done
with ``iptables`` (and its siblings ``ip6tables``, ``arptables``,
``ebtables``). Each protocol had a separate tool and a separate kernel module.
The configuration was verbose, error-prone, and architecturally limited.

**nftables (merged into the Linux kernel in 2014, Linux 3.13)** was designed
to replace all of them with a single unified framework:

+---------------------+---------------------------+-------------------------------+
| Aspect              | iptables                  | nftables                      |
+=====================+===========================+===============================+
| Tools               | ``iptables``,             | Single ``nft`` tool.          |
|                     | ``ip6tables``,            |                               |
|                     | ``ebtables``,             |                               |
|                     | ``arptables``             |                               |
+---------------------+---------------------------+-------------------------------+
| Protocol support    | Separate tables and       | ``inet`` family handles       |
|                     | rulesets for IPv4 and     | IPv4 and IPv6 in one          |
|                     | IPv6.                     | ruleset.                      |
+---------------------+---------------------------+-------------------------------+
| Rule representation | Kernel table +            | Native bytecode in kernel,    |
|                     | userspace tool sends      | compiled by userspace.        |
|                     | individual commands.      |                               |
+---------------------+---------------------------+-------------------------------+
| Atomic updates      | No — each rule change     | Yes — entire ruleset can be   |
|                     | is a separate operation.  | replaced atomically.          |
+---------------------+---------------------------+-------------------------------+
| Performance         | Linear match traversal.   | Set-based matching, maps,     |
|                     |                           | concatenations — O(1) for     |
|                     |                           | many cases.                   |
+---------------------+---------------------------+-------------------------------+
| Syntax              | Programmatic shell        | Structured, closer to a       |
|                     | commands.                 | configuration language.       |
+---------------------+---------------------------+-------------------------------+

**Current state:** Modern distributions ship with nftables as the default
kernel back-end. On many distributions, ``iptables`` commands are transparently
translated to nftables via the ``iptables-nft`` compatibility layer. If you
run ``iptables --version`` and see ``nf_tables``, you are already using
nftables under the hood.

::

    $ iptables --version
    iptables v1.8.9 (nf_tables)

**Recommendation:** If you are writing new firewall configurations, use
nftables natively. If you are maintaining legacy systems with existing
iptables rules, migrate incrementally — the compatibility layer handles both
simultaneously.

Simple Firewall Script Example
=====================================

Below is a complete, production-ready nftables configuration for a typical
Linux server (web server with SSH access):

.. code-block:: nft
   :caption: /etc/nftables.conf

    #!/usr/sbin/nft -f

    flush ruleset

    table inet filter {
        chain input {
            type filter hook input priority filter; policy drop;

            # Allow loopback
            iif "lo" accept

            # Allow established/related
            ct state established,related accept

            # Drop invalid
            ct state invalid drop

            # ICMP (ping) — ratelimit to prevent abuse
            icmp type echo-request limit rate 10/second accept
            icmpv6 type echo-request limit rate 10/second accept

            # SSH — allow from management office only
            ip saddr 203.0.113.0/24 tcp dport 22 accept

            # HTTP/HTTPS — allow from anywhere
            tcp dport { 80, 443 } accept

            # Log dropped packets
            log prefix "nftables-drop: " drop
        }

        chain forward {
            type filter hook forward priority filter; policy drop;
        }

        chain output {
            type filter hook output priority filter; policy accept;
        }
    }

    table inet nat {
        chain postrouting {
            type nat hook postrouting priority srcnat; policy accept;
        }
    }

Load it with: ``sudo nft -f /etc/nftables.conf`` or ``sudo systemctl restart nftables``.

Logging and Debugging
=============================

nftables logging uses the kernel's logging subsystem. Logged packets appear in
the system journal (view with ``journalctl -k``).

.. code-block:: nft

    # Log with a custom prefix
    log prefix "INPUT-DROP: " drop

    # Log with rate limiting (prevent log flooding)
    log prefix "INPUT-DROP: " limit rate 1/minute drop

    # Log to a specific syslog facility
    log prefix "FW: " level info

For debugging, use ``nft monitor`` to watch nftables events in real time:

.. code-block:: bash

    sudo nft monitor

This shows every rule, chain, and table addition or deletion as it happens —
invaluable when testing dynamic firewall configurations.
